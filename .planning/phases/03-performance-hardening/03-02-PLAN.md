<task type="auto">
  <name>Task 1: Refactor pdfService to Transfer File Objects</name>
  <files>services/pdfService.js</files>
  <action>
    Modify WorkerPool.startTask in pdfService.js to stop converting files to ArrayBuffers on the main thread.
    
    Changes:
    - Remove 'await file.arrayBuffer()'.
    - Send the 'file' object (which is a Blob) directly in the postMessage payload.
    - Note: Blobs are inherently transferable/clone-efficient in browser engines.
    - Increase MAX_FILE_SIZE_MB constant from 100 to 1024 (1GB) to allow larger files.
  </action>
  <verify>
    <automated>npm test tests/pdfService.test.js</automated>
  </verify>
  <done>WorkerPool sends File objects directly to workers; size limit increased to 1GB.</done>

<task type="auto">
  <name>Task 2: Implement WorkerFS in pdfWorker</name>
  <files>services/pdfWorker.js</files>
  <action>
    Update processFile in pdfWorker.js to use WORKERFS mounting instead of FS.writeFile.
    
    Implementation:
    - Create a mount point (e.g., '/mnt') if it doesn't exist.
    - Use 'qpdfModule.FS.mount(qpdfModule.WORKERFS, { files: [file] }, "/mnt")'.
    - Update callMain to read from the mounted path: '/mnt/' + file.name.
    - Wrap callMain in a try-finally block to ensure 'qpdfModule.FS.unmount("/mnt")' is always called.
    - Keep output processing in MEMFS (standard) but ensure immediate unlinking after reading results.
    - Remove magic-byte validation if it requires reading the whole file (or refactor to read only first 4 bytes via Blob.slice).
  </action>
  <verify>
    <automated>npm test tests/pdfWorker.test.js</automated>
  </verify>
  <done>Worker processes files via WorkerFS; memory usage is minimized during large file processing.</done>
</task>
